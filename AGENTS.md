# AGENTS.md

## Project Overview

`chain-monitor` is a Python-based on-chain monitoring, signal, archive, SQLite mirror, Telegram notification, and reporting system.

The main runtime listens to Ethereum activity, parses raw events, enriches them with pricing / LP / CLMM / market context, routes them through signal quality gates, records archives and SQLite mirrors, and emits user-facing Telegram messages only after strict state / opportunity filtering.

This project is a research and trading-assistance tool.

It is **not** an auto-trading system and must not be changed to:

- place orders
- connect to private trading accounts
- infer position sizing
- decide leverage
- produce direct buy / sell instructions
- bypass risk blockers

`CANDIDATE` is not a trade signal. `VERIFIED` is still not an order instruction.

---

## Core Architecture

Important modules:

- `app/main.py`: async runtime entrypoint.
- `app/listener.py`: block/log polling, monitored address and LP pool scan logic.
- `app/pipeline.py`: main processing pipeline from parsed event to signal, archive, state, opportunity, and delivery.
- `app/config.py`: environment-driven runtime configuration.
- `app/filter.py`: private address book, aliases, entity metadata, and watch address loading.
- `app/lp_registry.py`: LP pool book loading, validation, major-pool coverage, and runtime-supported chain guards.
- `app/market_context_adapter.py`: fixture / unavailable / live market context adapter.
- `app/trade_action.py`: LP and market context based trade-action interpretation.
- `app/asset_market_state.py`: asset-level state machine and Telegram suppression.
- `app/trade_opportunity.py`: final `CANDIDATE / VERIFIED / BLOCKED` opportunity layer.
- `app/sqlite_store.py`: SQLite mirror, migration, compact, retention, and integrity tooling.
- `app/quality_reports.py`: operational, quality, market-context, DB, and opportunity reports.
- `reports/generate_daily_report_latest.py`: canonical daily report generator.
- `reports/generate_daily_compare_report.py`: canonical daily compare generator.
- `reports/legacy/`: legacy/debug report scripts only.

Current user-facing stack:

1. Raw / parsed events
2. Signals and delivery decisions
3. LP asset cases
4. Quality / outcome stats
5. Trade action interpretation
6. Asset market state machine
7. Final trade opportunity layer
8. Telegram summaries and daily reports

Telegram is intentionally sparse. It should focus on state changes, risk blockers, candidates, and verified opportunity state changes. Full research flow belongs in archive, SQLite, JSON/CSV summaries, and reports.

## Working Rules For Agents

- Work from the repository root.
- For Hermes automation, default to read-only.
- For Codex/code-change tasks, code edits are allowed only when the user explicitly requests implementation.
- Do not modify code unless explicitly asked.
- Do not use `git add .`.
- Do not commit generated reports, SQLite files, archive files, cache files, private config, or private data.
- Keep changes localized to the requested behavior.
- Preserve existing contracts between listener, pipeline, processor, state, opportunity, archive, SQLite, reports, and notifier.
- Prefer existing helpers and managers over new abstractions.
- Use structured JSON/CSV/parsers instead of ad hoc string parsing when possible.
- Keep comments short and useful.
- When changing report schemas, update tests and keep JSON summaries machine-readable.
- When changing signal, state, opportunity, archive, SQLite, or Telegram behavior, add or update focused tests for the relevant Make target.
- If a report input is missing or schema drifts, write limitations / warnings instead of guessing.
- Keep behavior fail-closed when private config is missing.
- Do not fabricate watch addresses, LP pools, previous reports, outcomes, prices, or market context.
- Preserve the distinction between research signals and trading guidance.

Hard blockers such as the following must remain hard blockers:

- `NO_TRADE`
- `NO_TRADE_LOCK`
- `DO_NOT_CHASE_LONG`
- `DO_NOT_CHASE_SHORT`

Do not weaken opportunity gates casually. `VERIFIED` should require strong evidence such as live context, direction agreement, broader confirmation, quality thresholds, outcome history, risk filters, no conflict, and clear invalidation.

Important loading behavior:

- `address_aliases.json` can only enhance addresses already present in `addresses.json`.
- Alias-only entries must not create monitored addresses unless explicitly configured.

- Placeholder LP pools must remain `enabled=false`.
- Current active LP scanning is Ethereum-only.
- Solana and Base pools may appear in coverage diagnostics but must not be treated as Ethereum runtime scan targets.

## Hermes Automation Boundaries

Hermes may be used as a VPS-based report analyst and Telegram assistant.

Hermes should be treated as read-mostly automation.

Hermes may:

- read source code
- read `Makefile`
- read generated reports
- read non-sensitive diagnostics
- run approved read-only or report-generation commands
- write analysis files to `reports/hermes/`
- generate canonical report outputs under `reports/daily/` and `reports/daily_compare/` when running approved report-generation commands
- produce Telegram summaries
- create patch plans or Codex prompts when asked

Hermes must not:

- modify `.env`
- modify private config
- modify private address lists
- modify API keys, Telegram tokens, RPC credentials, or service secrets
- modify SQLite files directly
- modify archive files directly
- modify runtime caches directly
- start, stop, restart, or duplicate the production listener unless explicitly requested
- use `--yolo` mode for production VPS work
- run destructive maintenance commands without explicit approval
- convert this project into an auto-trading system

Allowed automatic commands for daily report analysis:

```bash
make env-check
make report-source-fast
make db-integrity DB_INTEGRITY_FAST=YES
make db-summary
make opportunity-db
make coverage
make report-daily
make report-daily-date DATE=YYYY-MM-DD
make daily-compare DATE=YYYY-MM-DD
make daily-compare-rebuild DATE=YYYY-MM-DD
```

Use `daily-compare-rebuild` only when canonical report inputs are missing or stale, and state that rebuild was performed.

Commands requiring explicit user approval:

```bash
make daily-close DATE=YYYY-MM-DD
make daily-close-strict DATE=YYYY-MM-DD
make db-migrate-date DATE=YYYY-MM-DD
make sqlite-checkpoint
```

Commands requiring exact command confirmation in the same conversation:

```bash
make run
make run-research
make db-compact-execute CONFIRM=YES
make db-vacuum CONFIRM=YES
make db-prune-execute CONFIRM=YES
make archive-compress-date DATE=YYYY-MM-DD CONFIRM=YES
make daily-close DATE=YYYY-MM-DD COMPRESS=YES CONFIRM=YES
```

If a command has a dry-run mode, prefer dry-run first.

Before running `make run` or `make run-research`, check whether a production listener is already running by using `ps`, `pgrep`, `systemctl status`, `tmux ls`, or the project’s known runtime status command. Do not duplicate listeners.

## VPS / Hermes Runtime Safety

When Hermes runs on the VPS, it should run as a dedicated low-privilege user.

Recommended permissions:

- read access to the repository
- read access to generated reports
- read access to non-sensitive diagnostics
- write access to `reports/hermes/`
- write access to `reports/daily/` and `reports/daily_compare/` only for approved canonical report-generation commands
- no write access to `.env`
- no write access to private keys
- no write access to production runtime config unless explicitly granted
- no permission to restart production services unless explicitly requested

Hermes should not use root privileges for normal report analysis.

If a command fails due to permissions, explain the missing permission and propose a minimal fix. Do not escalate permissions automatically.

## SQLite And Archive

SQLite is a mirror / query / analysis layer. It does not replace NDJSON archive or JSON caches.

Default DB:

```text
data/chain_monitor.sqlite
```

Archive base:

```text
app/data/archive
```

Reports default to DB-first reading, then archive/cache fallback when allowed.

Useful commands:

```bash
make db-init
make db-summary
make db-integrity
make db-report
make report-source-fast
make report-source
make db-migrate-date DATE=YYYY-MM-DD
make db-migrate-all
make sqlite-checkpoint
```

Maintenance order:

```bash
make db-backfill-payload-dry-run
make db-backfill-payload-execute CONFIRM=YES
make db-compact-dry-run
make db-compact-execute CONFIRM=YES
make db-vacuum CONFIRM=YES
make db-integrity
make db-summary
```

Safety distinctions:

- Archive compression gzips `.ndjson`; it does not delete data.
- SQLite compact removes duplicated full payload from SQLite only after archive/hash checks.
- Vacuum reclaims SQLite file space.
- Prune deletes rows and is not part of normal daily-close.
- Commands requiring `CONFIRM=YES` are intentionally dangerous or state-changing.
- Prefer dry-run first.

Do not delete or reset:

- `app/data/archive/`
- `data/*.cache.json`
- `data/chain_monitor.sqlite`
- SQLite WAL/SHM files
- generated historical reports

Treat `archive/cache/db` as valuable historical state. Old data is used for opportunity calibration, blocker analysis, quality scores, outcome settlement, recovery, and report fallback.

## Daily Close

Daily close flow:

```bash
make daily-close DATE=YYYY-MM-DD
make daily-close DATE=YYYY-MM-DD COMPRESS=YES CONFIRM=YES
make daily-close-strict DATE=YYYY-MM-DD COMPRESS=YES CONFIRM=YES
```

Rules:

- Do not compress the current UTC date while the listener may still be writing unless explicitly using `ALLOW_TODAY=YES`.
- `COMPRESS=YES` alone is not enough; actual compression still requires `CONFIRM=YES`.
- `daily-close` does not prune and must not delete archive/cache/db.
- For automation, prefer report generation over daily-close unless the user explicitly requested close/maintenance.
- Always explain what date is being closed and why that date is safe.
